## 一、项目概述

### 1.1 业务背景

物流票据处理场景中，司机卸货后需拍照留存两类图片：

| 类型 | 内容 | 特征 |
|------|------|------|
| **票据照片** | 销售出库单 / 发货单 / 计量凭证 | 含条形码、二维码、发货单号（4920 开头 10 位数字） |
| **卸货照片** | 车辆 / 货物卸货现场 | 无条形码 / 二维码 |

两类照片按拍摄顺序交替存放在同一文件夹（先票据、后卸货），需要：

1. 自动识别票据上的单号，将票据照片重命名为 `{单号}.jpg`
2. 将紧随其后的卸货照片重命名为 `{单号}_2.jpg`
3. 支持将"原始出库单号"批量替换为"新出库单号"
4. 支持登录账号后，自动将图片上传到对应业务节点
5. 上传成功后在本地归档记录

### 1.2 技术栈

| 组件 | 选型 | 说明 |
|------|------|------|
| 语言 | Python 3.9+ | 跨平台 |
| 条形码识别 | zxing-cpp（主）、pyzbar（兜底） | 多引擎互补 |
| 二维码识别 | zxing-cpp + OpenCV QRCodeDetector + pyzbar | 三层解码 |
| 图像预处理 | OpenCV + Pillow | CLAHE / 蓝色通道分离 / 多阈值 / 多尺度 |
| OCR 兜底 | Tesseract（chi_sim + eng） | 最终兜底提取数字 |
| CLI 框架 | argparse（子命令模式） | 轻量无额外依赖 |

### 2.3 核心处理流程（v1.0）

```
输入: 001-Pic/ 中的所有图片
        │
        ▼
┌─ 第一轮 · 快速扫描（多线程）──────────────────────┐
│  每张图 3 步:                                      │
│   ① 读取图片 + EXIF 修正                           │
│   ② 全图快扫（条形码 + 二维码，0°/180° 或 90°/270°）│
│   ③ 半图补扫（上半 / 下半各跑一次快扫）             │
│                                                    │
│  结果: ✅ 识别成功 → 移动到 识别正确/{码值}.jpg      │
│        ⏳ 未识别   → 进入第二轮                      │
│        ⚠️ 读取失败 → 移动到 未识别/                  │
└────────────────────────────────────────────────────┘
        │ (仅未识别的)
        ▼
┌─ 第二轮 · 深度扫描（多线程）──────────────────────┐
│  每张图 5 步:                                      │
│   ① 读取图片                                       │
│   ② 全图快扫                                       │
│   ③ 半图补扫                                       │
│   ④ 二维码深扫（ROI + 颜色分离 + CLAHE + 多尺度）   │
│   ⑤ OCR 兜底（Tesseract 提取数字）                  │
│                                                    │
│  结果: ✅ 识别成功 → 移动到 识别正确/{码值}.jpg      │
│        ❌ 识别失败 → 移动到 未识别/                  │
└────────────────────────────────────────────────────┘
```

### 2.4 码值仲裁规则

1. 只接受**纯数字**，且匹配 `49\d{8}` 模式（优先 `4920\d{6}`）
2. 条形码和二维码各自产生候选值，按 `weight` 评分排序
3. **条形码 + 二维码命中同一数字** → 优先采用（最高置信度）
4. 仅单侧命中 → 取分数最高的数字值
5. 均未命中 → OCR 兜底提取"发货单号"附近的数字

## 四、功能模块详细设计

### 4.1 模块 A — scan（扫描识别 + 原地重命名）

#### 改动点

| 变更项 | v1.0 | v2.0 |
|--------|------|------|
| 输入目录 | 硬编码 `001-Pic/` | `--input` 参数，默认 `./001-Pic` |
| 成功处理 | 移动到 `识别正确/` | **默认原地重命名**；`--move` 恢复旧行为 |
| 失败处理 | 移动到 `未识别/` | **默认保持原名不动**；`--move` 恢复旧行为 |
| 文件命名 | `{码值}.jpg` | `{码值}.jpg`（不变） |
| 配对命名 | 不支持 | `--pair` 模式下卸货照片命名为 `{码值}_2.jpg` |

#### 原地重命名逻辑

```python
# 伪代码
for image in sorted(input_dir.iterdir()):
    outcome = process_one(image)
    if outcome.status == "ok":
        new_name = f"{outcome.final_value}{image.suffix.lower()}"
        new_path = ensure_unique_path(image.parent, outcome.final_value, image.suffix)
        image.rename(new_path)
    else:
        pass  # 保持原名，不移动
```

#### 配对命名逻辑（`--pair` 模式）

```python
# 伪代码 — 按文件名排序后顺序处理
current_code = None

for image in sorted(input_dir.iterdir()):
    outcome = process_one(image)

    if outcome.status == "ok":
        # 票据照片 — 识别到码值
        current_code = outcome.final_value
        image.rename(image.parent / f"{current_code}{image.suffix}")

    elif current_code is not None:
        # 卸货照片 — 无码，跟随上一张票据
        suffix_idx = 2  # 默认 _2，多张卸货依次递增 _3, _4 ...
        target = ensure_unique_path(image.parent, f"{current_code}_{suffix_idx}", image.suffix)
        image.rename(target)

    else:
        # 开头就是卸货照片，前面没有票据 — 保持原名，输出警告
        print(f"⚠️ {image.name} 无法配对（前面没有已识别的票据）")
```

**边界情况**：

| 场景 | 处理方式 |
|------|---------|
| 连续多张票据（不同码值） | 各自独立命名，不产生 `_2` |
| 一张票据 + 多张卸货 | `{码值}_2.jpg`、`{码值}_3.jpg` ... |
| 开头就是卸货照片 | 保持原名 + 警告 |
| 连续多张票据（同一码值） | `{码值}.jpg`、`{码值}_1.jpg` ... |

### 4.2 模块 B — rename（批量换号）

#### 输入格式

**CSV 映射文件**（`mapping.csv`）：
```csv
原始单号,新单号
4920174528,5030284639
4920512130,5030399001
4920604114,5030715288
```

#### 三层查重校验

```
校验 ①: 原始单号唯一性
  映射表中同一原始单号不能出现多次（→ 多个新单号矛盾）
  发现冲突 → 该原始单号所有行跳过

校验 ②: 新单号唯一性
  映射表中同一新单号不能被多个原始单号映射
  发现冲突 → 涉及的所有行跳过

校验 ③: 文件系统冲突
  新单号对应的文件名不能与目录中已有文件撞名
  发现冲突 → 该行跳过
```

#### 处理流程

```
1. 读取 mapping.csv → Dict[原始单号, 新单号]
2. 三层查重 → 生成 valid_map + conflict_report
3. 扫描文件夹 → 找到所有 {原始单号}.jpg 和 {原始单号}_N.jpg
4. 对 valid_map 中的每条:
   ├── {原始单号}.jpg     → os.rename → {新单号}.jpg
   ├── {原始单号}_2.jpg   → os.rename → {新单号}_2.jpg
   └── {原始单号}_N.jpg   → os.rename → {新单号}_N.jpg
5. 输出报告:
   ✅ 成功: 12 组
   ⚠️ 冲突跳过: 3 组（附详情）
```

#### 接口定义

```python
# src/batch_rename.py

@dataclass
class RenameResult:
    original: str           # 原始单号
    target: str             # 新单号
    status: str             # ok | conflict_dup_original | conflict_dup_target | conflict_file_exists | file_not_found
    files_renamed: list[tuple[str, str]]   # [(旧文件名, 新文件名), ...]
    reason: str             # 冲突原因

def load_mapping(csv_path: Path) -> dict[str, str]:
    """读取 CSV 映射文件。"""

def validate_mapping(mapping: dict[str, str], target_dir: Path) -> tuple[dict[str, str], list[RenameResult]]:
    """三层查重校验，返回 (有效映射, 冲突报告)。"""

def execute_rename(valid_map: dict[str, str], target_dir: Path, dry_run: bool) -> list[RenameResult]:
    """执行重命名。"""
```

## 五、跨平台兼容方案

### 5.1 平台差异与对策

| 依赖项 | macOS | Windows | 对策 |
|--------|-------|---------|------|
| `DYLD_LIBRARY_PATH` | 需要设置才能找到 libzbar | 不适用 | `run.sh` 自动设置；`run.bat` 不需要 |
| `pyzbar` → `libzbar` | `brew install zbar` | 无系统包管理 | 项目内捆绑 `libs/win64/libzbar-64.dll` |
| `zxing-cpp` | pip 安装 | pip 安装（有 wheel） | 无需特殊处理 ✅ |
| Tesseract | `brew install tesseract` | 需手动安装 | `setup.bat` 引导安装；或捆绑便携版 |
| `os.dup2` stderr 抑制 | 正常工作 | 行为可能不同 | 加 `try/except` 保护 |
| 路径分隔符 | `/` | `\` | 已使用 `pathlib.Path` ✅ |

### 5.2 pyzbar Windows 兼容代码

```python
# src/config.py
import sys
import os
from pathlib import Path

def setup_platform():
    """启动时调用，配置平台特定依赖。"""
    if sys.platform == "win32":
        dll_dir = Path(__file__).parent.parent / "libs" / "win64"
        if dll_dir.is_dir():
            os.add_dll_directory(str(dll_dir))
            # 兼容旧版 Python
            os.environ["PATH"] = str(dll_dir) + os.pathsep + os.environ.get("PATH", "")
```

```python
@dataclass
class ScanOutcome:
    """单张图片的扫描结果。"""

    image_type: str = ""     # ticket | unload | unknown
    paired_with: str = ""    # 配对的码值（卸货照片用）
    ocr_hint: str = ""
    error_note: str = ""

@dataclass
class RenameMapping:
    """单条重命名映射。"""
    original: str
    target: str
    status: str              # valid | dup_original | dup_target | file_exists
    reason: str = ""

@dataclass
class UploadRecord:
    """单条上传记录。"""
    order_number: str
    ticket_uploaded: bool = False
    unload_uploaded: bool = False
    upload_time: str = ""
    node_id: str = ""
    error: str = ""
```

## 九、技术风险

| 风险 | 等级 | 应对措施 |
|------|------|---------|
| 票据/卸货照片顺序不固定导致配对错误 | 🟡 中 | 提供 `--dry-run` 预览；配对失败保留原名 |
| Windows 上 pyzbar DLL 加载失败 | 🟡 中 | pyzbar 仅作兜底，zxing-cpp 为主力；DLL 加载失败时 graceful 降级 |
| 大量图片并发上传导致被限流 | 🟢 低 | 控制并发数 + 重试机制 + 指数退避 |
| Token 过期中途上传中断 | 🟢 低 | 上传前检查 Token 有效期；逐张记录归档避免重复 |
| CSV 映射文件编码问题（GBK vs UTF-8） | 🟡 中 | 自动检测编码（chardet）或约定 UTF-8 |

## 十、附录

### A. 码值正则匹配规则

```python
# utils.py → normalize_numeric_value()
# 优先匹配 4920 开头 10 位
r"4920\d{6}"
# 其次匹配 49 开头 10 位
r"49\d{8}"
```

### B. 二维码深扫预处理流水线

```
原图 → 颜色分离（灰度 / Lab-B反转）
         → 背景扣除（高斯模糊51 减去）
             → 多尺度放大（2x / 3x / 4x）
                 → 二值化（OTSU / CLAHE+OTSU / 自适应高斯）
                     → 白边填充（24px）
                         → 解码（zxingcpp → cv2 → pyzbar）
```

### C. 交付压缩包清单

```
003_HD.zip
├── run.sh              ← Mac 运行
├── run.bat             ← Windows 运行
├── setup.sh            ← Mac 首次安装
├── setup.bat           ← Windows 首次安装
```
