# 项目指南

## 项目定位

这个项目当前的主交付形态是：

- `Mac + Windows` 双平台离线包
- `Windows EXE` 免 Python 打包链

目标用户不需要先理解 CLI，也不需要把命令安装到系统 `PATH`。推荐使用方式是：

1. 把图片放进 `input/`
2. Windows 运行 `start.bat`
3. macOS 运行 `start.command`
4. 首次运行时自动创建 `.venv/`
5. 完成后在原目录查看重命名结果

## 核心运行结构

核心运行文件：

- `src/pipeline.py`：主流程，负责扫描目录、识别、重命名
- `src/__main__.py`：CLI 入口
- `src/barcode.py`：条形码识别
- `src/qrcode_scan.py`：二维码识别
- `src/ocr.py`：OCR 兜底识别
- `src/config.py`：平台依赖路径和 `tessdata` 配置
- `tessdata/`：OCR 训练数据

离线包入口文件：

- `start.bat`：Windows 入口
- `start.command`：macOS 入口
- `setup.bat`：Windows 本地环境安装器
- `setup.command`：macOS 本地环境安装器
- `input/`：固定输入目录

Windows EXE 入口文件：

- `Release/exe_entry.py`：PyInstaller 入口，复用 `src.__main__.main()`
- `Release/build_windows_exe.py`：Windows exe 构建脚本
- `.github/workflows/build-windows-exe.yml`：GitHub Actions 构建
- `.github/workflows/release-windows-exe.yml`：GitHub Actions Release 发布

## 冗余代码分析

当前仓库里存在几类“不是离线包运行必需”的文件，但它们不是无意义冗余，而是保留的辅助层：

- `install.sh` / `install.ps1`
  作用：面向 CLI 安装场景。
  结论：对离线包运行不是必需，但对开发者和命令行安装仍有价值。

- `run.sh` / `run.bat` / `run.command`
  作用：手动指定目录运行。
  结论：和 `start.bat` / `start.command` 存在入口重叠，但保留作为调试/手动运行备用入口。

- `Release/build_windows_exe.py` / `.github/workflows/*.yml`
  作用：面向 Windows EXE 分发。
  结论：对源码本地运行不是必需，但对“免 Python 发布”是关键交付层，不属于冗余。

- `tests/`
  作用：回归测试。
  结论：对最终离线包运行不是必需，因此不进入 Release 包。

- `README.md`
  作用：项目总览。
  结论：适合保留在源码仓库和 Release 包中，但不是启动所必需的运行文件。

## Release 取舍原则

进入 Release 离线包的文件：

- 启动脚本
- 平台对应的安装脚本
- 使用说明
- 项目指南
- 运行源码
- OCR 数据
- 离线 `wheels/`
- 示例输入目录
- 版本信息

不进入 Release 离线包的文件：

- 测试代码
- Git 元数据
- 本地缓存和虚拟环境
- 旧数据目录和隐私目录

## 为什么不直接打包一个通用 venv

不直接这样做，原因有两个：

- `venv` 是平台相关的，Mac 和 Windows 不能共用
- `venv` 往往和创建时的绝对路径绑定，直接打包后在别的目录解压，容易失效

所以当前方案不是“打一个通用 venv”，而是：

- Windows 包自带 `win` 依赖 wheels，并由 `setup.bat` 本地创建 `.venv`
- macOS 包自带 `macos` 依赖 wheels，并由 `setup.command` / `setup.sh` 本地创建 `.venv`

这仍然属于“包内自带运行依赖”，但比直接搬运一个 venv 更稳。

## 当前运行边界

当前构建产物已经尽量做到“开箱即用”，但仍有几个现实边界：

- 离线包内已经包含离线 Python 依赖 `wheels/`
- 首次运行不需要联网下载这些依赖
- 目标 Windows 机器当前需要 `Python 3.10 x64`
- 目标 macOS 机器当前需要 `Python 3.10`
- macOS 离线 wheels 会跟随构建机架构；如果要给另一种 Mac 架构使用，需要在对应架构上重新构建
- 若没有安装 `Tesseract`，条形码/二维码主流程仍可运行，但 OCR 兜底能力受限
- Windows `exe` 包不需要目标机器安装 Python，但 GitHub Actions 或本地 Windows 构建机仍需要能安装 `PyInstaller` 和 `Tesseract`

## 维护建议

- 修改运行依赖后，同步更新 `requirements.txt`
- 修改版本号时，同步更新：
  - `src/__init__.py`
  - `pyproject.toml`
  - `CHANGELOG.md`
- 发布离线包前，执行：
  - `.venv/bin/python -m unittest discover -s tests -v`
  - `.venv/bin/python Release/build_portable.py`
- 发布 Windows EXE 前：
  - 在 Windows 上执行 `py -3.10 -m pip install -e .[build]`
  - 执行 `Release\build_windows_exe.bat`
  - 或推送到 GitHub，让 Actions 自动构建并发布
